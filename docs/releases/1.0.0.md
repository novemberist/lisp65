## What shipped

- Native MEGA65 Lisp REPL
- On-device `lcc` bytecode compiler
- Closures, macros, higher-order functions, and strict arity
- Full-screen Emacs-style editor
- On-demand IDE and optional IDEX feature tiers
- Source evaluation and persistent L65M library compilation
- Transactional M65D create/replace persistence
- Boot stager for the `L65SYS` product image
- A blank `L65WORK` convenience image; any valid non-product 1581 disk is valid

## IDE input erratum (2026-07-15)

A post-release end-to-end review found that the 1.0.0 hardware UX evidence
injects normalized key events after the physical keyboard boundary. It proves
the editor dispatch and backends on hardware, but not every documented physical
key chord. In particular:

- load IDE, IDEX, and M65D from `L65SYS` before swapping to a work disk; the
  immutable bundled `README-FIRST.txt` omitted IDEX and M65D from its example;
- the implemented command launcher is `C-x x` or `C-x Return`, requires IDEX,
  and is not a physical `M-x` binding;
- `C-Space` is unreachable because its zero code collides with the driver's
  empty-queue value, so mark/region keyboard workflows are broken;
- the remaining Control and `C-x` command backends are tested, but the physical
  key path is not accepted end to end and must be treated as experimental;
- persistent compilation requires a preallocated FASL target, while the supplied
  blank work D81 contains no `fasl0`--`fasl2` slots.

The live [User Guide](../user-guide.md) contains the corrected workflow and
workarounds. This erratum changes no 1.0.0 product artifact, hardware receipt,
tag, or release-bundle byte. Release [1.0.1](1.0.1.md) republishes the corrected
first-session instructions without changing product bytes; it deliberately
does not provision legacy FASL slots. The modifier-aware hardware input path
and transactional compiler persistence are planned for 1.1.

## Known limitations

- Release 1.0.0 is an early release for exploration and small projects with
  backups, not unattended production use or irreplaceable data.
- The released baseline leaves 120 symbol entries, 2,160 name-pool bytes, and
  32 L65M directory entries. Libraries and definitions are append-only, with no
  `unload`.
- The on-device compiler cannot yet produce a standalone runtime or bootable
  application disk.
- M65D and editor saves are limited to 8,192-byte payloads. Evaluator `load` has
  a separate 38,400-byte staging ceiling; the supported editable load/save
  round trip is 8 KiB.
- Xemu-only operation has limited fidelity. F011 writes, SD buffer mapping,
  Freezer behavior, reset behavior, and hardware timing still require a real
  MEGA65 for the release claims.
- The physical editor input path is not fully bound to the documented keymap.
  `C-Space` is broken, physical `M-x` is absent, and other Control chords are
  experimental in 1.0.0.
- The blank work D81 has no preallocated FASL target; persistent compilation
  reports `slot missing` unless the medium was provisioned externally.
- One-drive, two-media workflow
- No on-device disk formatter
- No undo/redo in the editor
- No full ANSI Common Lisp compatibility
- No atomicity claim for the measured Freezer race after a sector write has
  already started; at most one sector may cross the media boundary before the
  terminal guard stops subsequent writes
